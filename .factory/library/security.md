# Security

Security model, pairing expectations, and policy rules.

**What belongs here:** trust model, auth rules, approval gating, audit expectations, and fail-closed behavior.

---

- Pairing starts from a QR generated by the macOS shell.
- Tailscale protects the network path, but the product still owns pairing, identity, session auth, and authorization.
- Bridge identity must be device-unique and long-lived; deriving it only from the configured bind host/port is not sufficient for one-phone-to-one-Mac trust or identity-mismatch protection.
- Trust model: one phone to one Mac from day one.
- Pairing QR payloads are expected to carry a private bridge base URL; the current mobile validator rejects non-private routes with `privateRouteRequired`.
- For this mission only, a debug-only manual pairing helper may accept or derive an emulator-reachable bridge URL (default `http://10.0.2.2:3110`) strictly for Android emulator validation. This must not change release QR/Tailscale pairing semantics or weaken the existing trust-review/finalize contract.
- The canonical pairing startup path must verify an existing `tailscale serve` mapping to the localhost bridge port or successfully provision `tailscale serve --bg <bridge-port>` before advertising the MagicDNS HTTPS pairing base URL; if that mapping cannot be verified or later disappears, `/pairing/session` must fail closed instead of issuing QR sessions.
- Trust is only complete after bridge-side finalize succeeds and returns a bridge-owned `session_token`; local mobile storage alone is not authoritative trust.
- Authorization modes:
  - read-only
  - control-with-approvals
  - full-control
- Dangerous actions must be gated and auditable.
- Reconnect must fail closed if trust is revoked or bridge identity no longer matches stored trust, using the bridge handshake inputs `phone_id`, `bridge_id`, and `session_token`.
